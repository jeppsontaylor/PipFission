Below is the version I would actually build. The biggest change is that I would **not** let an optimizer freely choose arbitrary “perfect” buy/sell points over the full 1,000-point history. That is almost guaranteed to create an oracle target that looks brilliant in-sample and fails live. I would restrict the optimizer to a **causal label family**, tune it only inside training folds, generate **out-of-fold classifier probabilities**, then tune the trader only on those out-of-fold probabilities.

This is educational/system-design guidance, not individualized financial advice. Automated trading systems can fail because their assumptions stop matching the market; SEC investor materials specifically warn that automated tools may rely on assumptions that become wrong, and FINRA has warned about exaggerated claims around AI/auto-trading services. ([Investor.gov][1])

---

## 1. Core recommendation

Your idea is directionally strong, but the safest high-performance version is:

**Do not train the classifier to imitate globally optimized trades.**
Train it to predict **local, volatility-aware, path-dependent labels** generated by a constrained labeling rule such as triple-barrier labeling, then train a separate trader that decides when confidence is high enough to act.

The best architecture is:

```text
Raw OHLCV / tick / macro / cross-asset data
        ↓
Rust feature engine, point-in-time and lagged
        ↓
Event sampler: which bars are eligible candidate decisions?
        ↓
Fold-local label optimizer:
    triple-barrier / oracle-policy parameters only
        ↓
Classifier model zoo:
    LightGBM, XGBoost, CatBoost, logistic, RF/ExtraTrees, small MLP
        ↓
Purged / embargoed out-of-fold probabilities for every train point
        ↓
Trader optimizer:
    thresholds, stop loss, take profit, min/max hold, sizing, cooldowns
        ↓
Locked 100-point holdout test
        ↓
Production Rust inference + Rust trader
```

The validation change is crucial. Standard random k-fold is inappropriate for time-ordered data because it can train on future data and evaluate on past data; even scikit-learn’s `TimeSeriesSplit` exists specifically to preserve time ordering. ([Scikit-learn][2]) For financial labels with overlapping future horizons, I would go further and use **purged/embargoed CV** or **Combinatorial Purged Cross-Validation**. Recent work on backtest overfitting emphasizes that financial data has non-stationarity, autocorrelation, and regime shifts, and reports CPCV as stronger for mitigating overfitting than ordinary k-fold and walk-forward in their test setup. ([SSRN][3])

## 2. The labels I would use

I would use two labels, not one.

### Label A: side label

This is the binary classifier you asked for:

```text
1 = long / buy candidate
0 = short / sell candidate
```

But this label should only exist on **candidate event rows**, not every bar. Rows with no meaningful event should not dominate the classifier.

Use triple-barrier labeling:

For each candidate timestamp `t`:

```text
vol_t = EWMA volatility at t

upper barrier = entry_price_t * (1 + pt_mult * vol_t)
lower barrier = entry_price_t * (1 - sl_mult * vol_t)
vertical barrier = t + H bars
```

Then:

```text
side_label = 1 if upper barrier is hit before lower barrier
side_label = 0 if lower barrier is hit before upper barrier
drop or neutralize if neither barrier is hit before H
```

Triple-barrier labeling is a better fit than fixed-horizon up/down labels because it uses an upper barrier, lower barrier, and vertical time barrier, and it can set barriers dynamically from volatility rather than treating every observation as having the same risk. ([mlfinpy.readthedocs.io][4])

### Label B: meta/action label

This is the “should we actually trade?” model:

```text
1 = take the trade
0 = skip the trade
```

This is where no-trade belongs. Meta-labeling separates the side decision from the decision of whether to take/size the bet; the secondary model is trained to decide whether to take or pass on the primary signal. ([mlfinpy.readthedocs.io][4])

In production, the trader receives:

```text
p_long
p_short = 1 - p_long
p_take
```

Then it decides:

```text
go long  if p_long  > threshold_long  and p_take > threshold_take
go short if p_short > threshold_short and p_take > threshold_take
do nothing otherwise
```

### About your 30–50% balance requirement

I would **not** force 30–50% live trades. That is likely too high for many markets. I would enforce balance only in the **training labels**:

```text
side classifier:
    long_share among candidate events should be roughly 30–70%
    ideally closer to 40–60%

meta classifier:
    take_share should be large enough to learn, often 20–50%
```

If you literally need “30–50% buy/sell signals,” interpret that as:

```text
30–50% of rows become candidate event rows
and within those event rows, long/short is not badly skewed
```

Do not label every other row as “hold” and train a normal binary classifier on `buy vs hold`. That creates a majority-class classifier that learns not to trade.

## 3. Fixing the “optimal labels” idea

The dangerous version is:

```text
Use all 1,000 points → find perfect buy/sell entries → train classifier → k-fold score
```

That leaks information because the label set itself was optimized using the full future path.

The safer version is:

```text
For each outer fold:
    Use only the training portion to tune label parameters
    Apply those label parameters to that fold’s validation portion
    Train only on purged training rows
    Predict only validation rows
```

The optimizer may choose the **labeling parameters**, not arbitrary labels:

```text
event_threshold
vertical_horizon H
take_profit_multiple
stop_loss_multiple
min_return
volatility lookback
label neutralization rule
class-balance penalty
```

The label optimizer objective should be something like:

```text
maximize:
    median_inner_cv_sortino
    - penalty_for_low_trade_count
    - penalty_for_class_imbalance
    - penalty_for_high_turnover
    - penalty_for_large_drawdown
    - penalty_for_parameter_instability
```

A practical objective:

```text
score =
    median(Sortino across inner folds)
    - 0.50 * max_drawdown_penalty
    - 0.25 * turnover_penalty
    - 0.25 * class_imbalance_penalty
    - 0.25 * low_trade_count_penalty
```

You can still run an “oracle optimizer” for research, but I would use it only as a diagnostic. If you let it choose arbitrary entry/exit points, it will learn the path, not the market.

## 4. Cross-validation design

With 1,000 development points and 100 locked holdout points:

```text
points 0–999     = model development
points 1000–1099 = locked holdout
```

Do **not** touch points `1000–1099` until everything is frozen.

For development:

```text
Outer validation:
    CPCV or purged blocked k-fold

Inner validation:
    purged/embargoed CV for label params, model params, feature selection

Embargo:
    at least max_label_horizon bars
```

For each outer fold:

```text
1. Build features using only information available at or before t.
2. Tune label parameters on training fold only.
3. Generate labels for train/validation using those label parameters.
4. Purge train rows whose label horizon overlaps validation rows.
5. Embargo rows after validation if their future horizon could leak.
6. Tune classifier in inner CV.
7. Calibrate probabilities in-fold.
8. Predict validation probabilities.
9. Store out-of-fold probabilities.
```

At the end, every row in `0–999` should have an out-of-fold confidence number:

```text
timestamp
features_hash
true_side_label
p_long_oof
p_short_oof
p_take_oof
fold_id
label_params_used
model_id
```

This is the dataset used to tune the trader.

## 6. Classifier model zoo

I would test many models, but I would not let the search become huge because 1,000 points is small.

### Baselines

```text
regularized logistic regression
elastic-net logistic regression
linear SVM with calibrated probabilities
```

These are sanity checks. If complex models beat them only in-sample, reject the complex models.

### Main candidates

```text
LightGBM
XGBoost
CatBoost
RandomForest
ExtraTrees
small MLP
```

For tabular financial features, I would expect the champion to often be **LightGBM, XGBoost, CatBoost, or regularized logistic regression**.

LightGBM is designed as an efficient tree-based gradient boosting framework with parallel/distributed/GPU support, and it also has a C API that can be called from Rust-side production code. ([LightGBM Documentation][5]) XGBoost supports probability-style binary classification objectives such as `binary:logistic`, and CatBoost exposes `predict_proba` for class probabilities. ([XGBoost Documentation][6])

### Metrics to report

You asked for out-of-sample accuracy. I would report it, but not optimize for it.

Classifier report:

```text
OOF accuracy
OOF balanced accuracy
ROC-AUC
PR-AUC
F1
MCC
log loss
Brier score
calibration slope
precision at trading threshold
recall at trading threshold
fold-by-fold mean and std
```

Trader report:

```text
net return after costs
Sharpe
Sortino
Calmar
max drawdown
profit factor
expectancy per trade
hit rate
average win / average loss
turnover
average holding period
number of trades
exposure
tail loss / CVaR
capacity estimate
```

A classifier with mediocre accuracy can still be useful if its top-confidence predictions are profitable. A classifier with high accuracy can be useless if it is only right on tiny moves and wrong on large moves.

## 7. Trader optimizer

The trader should be optimized only on **out-of-fold classifier probabilities**, not in-sample predictions.

### Trader parameters to tune

| Category           | Parameters                                                                                    |
| ------------------ | --------------------------------------------------------------------------------------------- |
| Entry thresholds   | `long_threshold`, `short_threshold`, `take_threshold`, `min_prob_edge`, `min_expected_return` |
| Exit thresholds    | `exit_prob_threshold`, `reversal_exit_threshold`, classifier confidence decay                 |
| Stops              | `stop_loss_atr`, `stop_loss_pct`, volatility stop, catastrophic stop                          |
| Profit-taking      | `take_profit_atr`, `take_profit_pct`, partial take-profit, trailing stop                      |
| Holding rules      | `min_hold`, `max_hold`, time stop, session close exit                                         |
| Sizing             | fixed fractional, volatility targeting, capped Kelly, max position size                       |
| Portfolio risk     | max gross exposure, max net exposure, max correlated positions, sector caps                   |
| Trade throttles    | cooldown after loss, max trades per day, max turnover, spread/liquidity filter                |
| Regime filters     | only trade in allowed volatility/trend/liquidity regimes                                      |
| Risk kill-switches | daily loss limit, weekly loss limit, max drawdown stop, stale-data stop                       |
| Execution          | market/limit choice, limit offset, slippage model, commission model, borrow fees              |

### Trader objective

I would optimize for robust Sortino or deflated Sharpe, not plain Sharpe alone.

Example:

```text
score =
    median_CPCV_sortino
    - 0.75 * max_drawdown
    - 0.50 * CVaR_95
    - 0.25 * turnover
    - 0.25 * parameter_instability
    - 0.50 * low_trade_count_penalty
```

Constraints:

```text
min_trades >= 25 across development sample
max_drawdown <= allowed limit
turnover <= allowed turnover
average_spread_cost <= allowed cost
no single trade contributes > X% of total PnL
no single fold contributes > Y% of total PnL
```

If the best strategy only makes money in one fold, reject it.

## 8. Optimizers

For the Python training layer, use Optuna. It supports sampler-based hyperparameter optimization; its `Study` object chooses hyperparameter values from distributions, and the default sampler is TPE. ([Optuna][7])

I would use:

```text
Label optimizer:
    Optuna TPE or CMA-ES
    optionally NSGA-II for multi-objective return/drawdown/turnover

Classifier optimizer:
    Optuna TPE
    early stopping for boosting models
    nested purged CV

Trader optimizer:
    Optuna TPE + random/Sobol warmup
    optionally evolutionary search for mixed discrete/continuous rules
```

Use a walk-forward sanity check even if CPCV is the main optimizer. CPCV is great for robustness, but a trader also needs to survive a strict chronological simulation.

## 10. Concrete training loop

Pseudo-code:

```python
raw = load_data()
features = rust.make_features(raw)

dev = features.iloc[:1000]
holdout = features.iloc[1000:1100]

oof = []

for outer_fold in rust.cpcv_splits(dev, n_groups=10, test_groups=2, embargo=max_horizon):

    train_idx, valid_idx = outer_fold

    # 1. Label optimization only on train
    best_label_params = optuna.optimize(
        lambda trial: label_objective(
            trial,
            dev.iloc[train_idx],
            inner_cv="purged"
        )
    )

    # 2. Generate labels using fold-local params
    y_train = rust.make_labels(dev.iloc[train_idx], best_label_params)
    y_valid = rust.make_labels(dev.iloc[valid_idx], best_label_params)

    # 3. Purge overlaps
    train_idx_purged = rust.purge_overlaps(train_idx, valid_idx, y_train)

    # 4. Model zoo
    best_model = train_best_classifier(
        X=dev.iloc[train_idx_purged],
        y=y_train.loc[train_idx_purged],
        cv="inner_purged",
        models=["logistic", "lightgbm", "xgboost", "catboost", "extratrees"]
    )

    # 5. Calibrate probabilities inside fold
    calibrated = calibrate_in_fold(best_model)

    # 6. OOF probabilities
    probs = calibrated.predict_proba(dev.iloc[valid_idx])
    oof.append((valid_idx, probs, y_valid, best_label_params, best_model.id))

oof_probs = concat_oof(oof)

best_trader = optuna.optimize(
    lambda trial: rust.backtest_trader(
        oof_probs,
        dev_prices,
        trader_params_from_trial(trial),
        costs=realistic_costs
    )
)

# Freeze everything
final_label_params = aggregate_or_refit_label_params_without_holdout()
final_model = train_final_model(dev, final_label_params)
final_trader = best_trader

# Locked test
holdout_probs = final_model.predict_proba(holdout)
holdout_report = rust.backtest_trader(
    holdout_probs,
    holdout_prices,
    final_trader,
    costs=realistic_costs
)
```

The 100-point holdout is a **gate**, not proof. If the strategy fails there, do not tune on those 100 points and call it fixed. That turns the holdout into another validation set.

## 11. Production deployment

Production should deploy:

```text
1. final trained classifier
2. final probability calibrator
3. final trader parameters
4. exact feature schema
5. exact cost/slippage assumptions
6. risk limits
```

Live loop:

```text
on new bar/tick:
    update feature state in Rust
    compute point-in-time features
    run classifier inference
    calibrate probability
    pass probability to trader
    trader checks position/risk/cost/regime rules
    order manager sends order or does nothing
    risk engine monitors exposure and kill-switches
    log every decision, including no-trade decisions
```

Every decision should be logged:

```text
timestamp
features
p_long
p_take
position
chosen action
reason code
trader params version
model version
order result
realized slippage
```

No-trade logs are essential. Without them, you cannot diagnose threshold drift.

## 12. What I would freeze as the “best possible” strategy

My final recommended design:

```text
Labeling:
    event-based triple-barrier side labels
    plus meta-label for take/skip

Balance:
    train side classifier only on candidate events
    enforce long/short share roughly 40–60, minimum 30–70
    use sample weights rather than distorting live trade frequency

Validation:
    purged/embargoed CPCV for development
    strict 100-point chronological holdout
    no random k-fold

Classifier:
    LightGBM champion candidate
    XGBoost/CatBoost challengers
    regularized logistic regression baseline
    calibrated probabilities required

Trader:
    optimized on OOF probabilities only
    objective = robust Sortino/Sharpe minus drawdown, tail-risk, turnover, and instability penalties
    includes thresholds, stops, take-profit, min/max hold, sizing, cooldown, liquidity filters, kill-switches

Implementation:
    Rust for features, labels, CV, backtest, trader, execution
    Python only for ML training and Optuna orchestration
    deploy inference in Rust through LightGBM C API or ONNX Runtime
```

The most important principle is this:

**The classifier should never see in-sample optimized confidence, and the trader should never be optimized on in-sample classifier predictions.**

Use fold-local labels, out-of-fold probabilities, realistic costs, purging, embargo, and a locked chronological holdout. That gives you a system that has a chance of surviving production instead of only discovering the best way to overfit 1,000 points.
